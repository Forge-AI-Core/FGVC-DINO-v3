# Preprocess Pipeline: FGVC-DINO-v3 데이터 전처리 파이프라인

본 디렉토리는 FGVC-DINO-v3 모델 학습을 위해 구축된 **데이터셋 빌드 파이프라인**을 포함하고 있습니다. 원본 4K 이미지와 어노테이션 파일(`annotations.json`)을 기반으로 크롭, 클러스터링, 라벨 교정 및 데이터셋 분할을 자동화합니다.

## 🎯 핵심 해결 과제 (Core Objectives)

이 파이프라인은 원본 데이터셋이 지니고 있는 세 가지 치명적인 문제를 해결하기 위해 설계되었습니다.

### 1. 중복 객체에 의한 Data Leakage 방지 (Clustering)
원본 이미지를 촬영할 때 트럭을 비우지 않고 연속해서 촬영하였기 때문에, 적게는 2~3개부터 많게는 10개 이상 동일한 중복 객체가 여러 사진에 걸쳐 등장합니다. 
이를 단순 비율로 분할할 경우, 동일한 객체가 Train과 Val(또는 Test)에 모두 섞여 들어가는 심각한 **데이터 누수(Data Leakage)** 현상이 발생합니다.
* **해결책:** DINOv3 기반 임베딩 클러스터링을 적용하여 같은 객체들을 하나의 클러스터(Cluster Folder)로 묶습니다. 이후 개별 이미지가 아닌 **클러스터 폴더 레벨**에서 Train/Val/Test 분할을 수행하여 중복 객체의 데이터 누수를 원천 차단합니다.

### 2. 다수결(Majority Voting)을 통한 오탐 라벨 자동 교정
데이터셋 내부에는 잘못 라벨링된 오탐 데이터들이 섞여 있으나, 도메인 데이터셋에 대해 위험상황을 정의할 수 있는 기준과 도메인지식에 대한 저의 한계로 전수조사하는 주관적인 수동 재라벨링이 불가능했습니다.
* **해결책:** 동일한 객체로 묶인 클러스터 폴더 내에서 **지배적인 빈도수의 라벨(Dominant Label)**이 소수 라벨을 덮어씌우도록(Cascade Consensus) 설계했습니다. 이를 통해 동일 객체의 라벨이 하나로 통일되며 자연스럽게 오탐이 교정됩니다.
만약 라벨의 빈도가 동률을 이뤄 충돌할 경우 해당 클러스터 폴더는 탈락시키도록하여 데이터규모의 희생을 다소 감수하더라도 새로운 불완전한 오탐라벨이 발생하지 않도록 방지하였습니다.

### 3. 두 가지 버전의 데이터셋 빌드 (Unique vs Vanilla)
위의 전처리 과정을 거친 후, 실험 목적에 따라 두 가지 형태의 데이터셋을 동시에 출력합니다.
* **Unique Sampling 셋:** 각 클러스터 폴더에서 대표 샘플 1개만 추출하고 나머지는 탈락시켜 완전한 중복 제거를 이룬 데이터셋.
* **Vanilla 셋:** 분할된 Train/Val 구조 내에서 클러스터 안의 모든 샘플을 다시 흩뿌려 모은 데이터셋 (Data Leakage는 이미 클러스터 레벨 분할로 해결되었으므로 안전함)

## 📂 파이프라인 버전 안내

프로젝트 진행 상황에 따라 두 가지 버전의 분할 파이프라인이 존재합니다:

1. **`select_dominance_label/` (8:1:1 분할)**
   * 초기에 테스트셋이 확보되지 않았을 때 임시로 자체 검증을 수행하기 위해 Train 8, Val 1, Test 1 비율로 분할하는 원본 스크립트입니다.
2. **`select_dominance_label_train_val_8_2/` (8:2 분할)**
   * 기업으로부터 실제 테스트셋을 제공받은 이후 사용되는 스크립트입니다. 별도의 Test 추출 없이 Train 8, Val 2 비율로 분할하며, 결과물은 `set_with_testset` 경로에 저장됩니다. (제공받은 테스트셋은 수동으로 이식하였습니다)

## ⚠️ 실행 전 주의사항 (Warnings & Prerequisites)

* **저장 공간:** 파이프라인 실행 시 원본 이미지 크롭 및 클러스터링 폴더 복사 과정이 수반되므로 **최소 2GB 이상**의 여유 디스크 용량이 필요합니다.
* **필수 입력 파일 구조:**
  타겟 데이터 폴더 경로(예: `data/Iron-Scraps/my_classification` 혹은 `data/Iron-Scraps/set_with_testset`) 하위에 다음 요소가 반드시 존재해야 합니다:
  1. `originals/` 폴더 (원본 4K 이미지 보관)
  2. `annotations.json` (COCO 포맷 어노테이션 파일)
* **Background & Unknown 라벨 삭제:**
  1. 이전 태스크의 잔재로 인해 디텍션 모델이 `background_bg`라는 라벨을 생성해 둔 상태입니다.
  2. 다수결 교정 과정에서 빈도수가 완벽하게 동률일 경우, 해당 클러스터 폴더는 탈락 처리되어 `unknown` 라벨을 갖게 됩니다.
  * **조치사항:** 파이프라인 빌드가 모두 끝난 후, 출력된 데이터셋 폴더 내에서 **`background_bg`와 `unknown` 폴더는 반드시 수동 혹은 명령어로 전량 삭제**한 뒤 모델 학습(optuna 등)을 진행해야 합니다.

## 🚀 실행 방법

파이프라인은 25%, 50%(또는 10%), 0% 등 서로 다른 여백(Padding) 비율을 가진 데이터셋을 순차적으로 자동 생성합니다. 환경에 맞게 소스코드 내부의 `CLS_DATA_DIR` 경로 상수를 조정한 후 실행하세요.

```bash
# 8:1:1 자체 테스트셋 구축용 파이프라인
uv run python3 preprocess_pipeline/select_dominance_label/main.py

# 8:2 실제 테스트셋 도입용 파이프라인
uv run python3 preprocess_pipeline/select_dominance_label_train_val_8_2/main.py
```
